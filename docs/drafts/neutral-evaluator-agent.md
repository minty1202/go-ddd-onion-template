## 背景

2026-04-28 の usecase 層パッケージ命名議論で有用性を再確認。assistant が主観的に「(1) を推奨」と推した直後に、ユーザーから「その意見本当に中立？」と問われ、中立評価エージェントを挟むことで主観バイアスを排除できた。

## 何のために使ったか

assistant の **主観的バイアスを排除** し、観点ごとの長所・短所を客観的に整理するため。具体的に:

- assistant がユーザーの主張に同調しすぎる傾向への対処
- 設計判断で「どっちが筋か」が拮抗する場面で、中立な観点別評価が必要なとき
- 一次ソース（原典・著名 OSS）を引用させ、推測ベースの主張を排除するため

## どう使ったか

設計判断の論点ごとに、選択肢 A / B / C を **中立に** 評価させる形で投げた。今セッション中の使用例:

1. **usecase 層構造の選択**（サブパッケージ化 vs ファイル分割）— 観点別に拮抗、明確な勝者なしと判明
2. **パッケージ命名スタイル評価**（`define.NewUseCase` + `define.UseCase` vs `todo.NewDefineUseCase`）— 観点別評価で判断材料を提供

## どう効果があったか

- assistant の主観的「推奨」が **観点別評価で拮抗** と判明し、ユーザーが自分の好み（Rails 経験者の構造観など）を考慮して選ぶ判断材料になった
- 「これが正しい」という押し付けではなく、「観点ごとにこう評価できる」という形で **判断責任をユーザーに戻す** 効果
- 一次ソース引用を強制することで、推測ベースの主張が減った

## プロンプト構造のポイント

中立評価エージェントを呼ぶ際、必ず含めるべき要素:

1. **「結論を支持も否定もしない姿勢」を明示** — 「中立に評価してください」「これが正しいではなく、観点ごとに評価できる、を提示」
2. **判断軸の優先順位を明示** — このプロジェクトでは:
   - kgrzybek/modular-monolith-with-ddd（C# 軸）
   - VaughnVernon/IDDD_Samples（Java 原典）
   - Go 標準ライブラリ・著名 OSS の実運用プロジェクト
   - Go DDD テンプレートは補助参照のみ
   - ThreeDotsLabs/wild-workouts は参考にしない（除外）
3. **観点を分けて評価を要求** — DDD 文脈 / 言語特性 / 読みやすさ / 拡張性 等
4. **一次ソース引用を要求** — リポジトリ名 + URL + ディレクトリパス、または公式ドキュメントの URL
5. **「確認できなかった点は素直に書く」と明示** — 推測で埋めさせない

## 監査エージェント（`docs/drafts/audit-agent.md`）との関係

- 監査エージェント = 解釈の妥当性を中立に評価
- 中立評価エージェント = 設計判断の選択肢を観点別に中立評価
- 両者は概念的に近い。後で統合 or 整理する候補
