# 監査エージェントの使い方（TODO: 体系化）

## 状態

まだルール化されていない。後で体系化する前提のメモ。

## 背景

2026-04-17 の feature-first / layer-first 議論で有用性を確認した手法。
調査エージェント → 検証エージェント → 原典一次確認エージェント → **解釈の監査エージェント**、という多層の使い分けが効いた。

## 役割

調査結果や**解釈の妥当性**を中立に評価する独立エージェント。
結論を支持も否定もしない姿勢で、以下を評価する：

- 引用・データの読み取りが正確か（文脈省略、誤読）
- 論理の飛躍がないか
- 反対解釈の可能性はないか
- 根拠の強さ（一次 / 二次、サンプル数、文化バイアス）

## 他のエージェントとの違い

- 調査エージェント: データ・事実を収集する
- 検証エージェント: データの信頼性を評価する
- **監査エージェント: 解釈の妥当性を評価する（意味論のレイヤー）**

## いつ使うか（現時点の観察）

- 自分（または他のエージェント）の解釈を独立視点で検証したいとき
- アーキテクチャ決定など、結論の分岐点になる判断のとき
- 一次ソースに直接あたれない / あたりにくいテーマで、引用解釈の偏りを検証したいとき

## 組み方の原則（暫定）

- 結論を支持する言い方で渡さず、引用と解釈だけ中立に渡す
- 「支持も否定もしない姿勢で評価してください」と明示する
- 反対解釈の可能性を明示的に求める（「他の読み方はないか」）
- 5 段階評価などで結論を数値化させると判断が明確になる

## 今後の体系化で整理したい観点

- 調査 → 検証 → 監査 の多層サイクルの設計
- どのタイミングで監査を挟むか（毎回か、分岐点だけか）
- 監査結果の取り扱い（再調査 / 判断保留 / 追加論点）
- 監査エージェントへの依頼フォーマットの標準化（渡すべき情報、評価観点）
- 監査エージェントの仕事の定義をより明確にする（何を「仕事」とし、何を「仕事外」とするか）
