# コードに関するルール

- Go / proto / SQL のコードを書かない。コードの提示もしない
- アドバイスする際は、このプロジェクト（Todo アプリ）の要件ではなく、全く別の要件を仮定した例で説明する
  - 例: 「Todo を Done にする振る舞い」について聞かれたら、「注文を発送済みにする」のような別ドメインに置き換えて考え方を説明する
  - 架空例はあくまで説明手段。判断の根拠としては実プロジェクト・原典・公式ドキュメントを使う

## 例外: Go の具体的な書き方を学ぶためのコード

「Go ならどう書くのか」を学ぶための参考実装を求められた場合は、プロジェクト（Todo ドメイン）のまま素直に書く。

- 理由: import パスやツール呼び出しが架空になると、具体的な書き方の学習にならない
- 提供形式: ユーザーが自分の手で書き写して身につけるための参考コード。品質は実務でそのまま通用するレベルで提供し、学習用だからと簡略化しない
- 対象の目安:
  - Go のテストの書き方（testing パッケージ、テーブル駆動、モック、assert ライブラリの使い方など）
  - ツール・ライブラリの具体的な使い方（sqlc、goose、validator など）
- 対象外（従来通り別ドメインで説明する）:
  - ドメインロジック・設計判断の考え方（値オブジェクト、集約境界、ユースケース分割など）
