# アドバイスに関するルール

- 意見を求められたときは、Go の一般的な慣習やプロジェクトで使われている方法に基づいて答える
- 不明な場合は必ず調べ、根拠のある回答をすること
- ベストプラクティスを探すこと
- 会話の前の文脈に引っ張られず、先入観なく広く調査すること
  - 例: DDD について話していても、DDD に限定せず、Go の一般的なプロジェクトや Web アプリとしてのベストプラクティスも含めて探す
- 調査はサブエージェントにバックグラウンドで委任し、会話を止めないこと
- 根拠として信頼できるソース:
  - Go 公式ドキュメント / Go Blog
  - Go チームメンバーの発言
  - 有名な OSS プロジェクトの実装
  - Star 数が多くメンテされているリポジトリのパターン

## アドバイスを簡略化しない

このリポジトリは将来、新規 Go アプリの土台として clone される前提。書く / 推薦するものは **新規本番プロジェクトの初期コード** として扱う。

以下のような **このプロジェクト固有の状況を盾にした簡略化** はしない:

- 「サンプル実装だから / Todo アプリだから」
- 「単一クライアント前提だから / PoC だから」
- 「今回のスコープでは不要 / ADR でスコープ外」
- 「現在の規模では over-engineering」
- 「学習目的なので簡略でよい」

これらの言い回しが浮かんだ時点で、そのアドバイスは新規本番プロジェクトに「省略していいもの」として伝染する。**この言い方が口元に来た時点で違反**。

説明・推薦は **本番の Go プロジェクトで標準的にどう扱うか** から組み立てる。ADR でスコープ外とされている論点も次の順序で提示し、3 だけで終わらせない:

1. 本番の標準形（Vernon IDDD、Spring Data、ent、Hibernate などの実装に基づいて）
2. このプロジェクトの現在地（実装済み / 決定済みの範囲）
3. 未決部分（別 ADR で扱う TODO）
